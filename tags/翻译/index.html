Nginx手册阅读笔记 —— 管理Nginx

URL： [http://nginx.org/en/docs/control.html]

nginx 可以通过信号来控制。 主进程的进程 ID 默认会被写入文件 /usr/local/nginx/logs/nginx.pid 。 文件名可以在编译时(configure) 或 通过 nginx.conf 中的 pid 命令来更改。 主进程支持以下几种信号：

信号 行为
TERM, INT 快速关闭
QUIT 优雅关闭
HUP 重新加载配置，用新的配置启动新的进程，优雅的关闭旧的进程
USR1 重新打开log文件
USR2 升级可执行文件
WINCH 优雅关闭worker进程

也可以通过信号来控制独立的工作进程，尽管它不是必须的。 支持的信号如下：

信号 行为
TERM,INT 快速关闭
QUIT 优雅关闭
USR1 重新打开log文件
WINCH 非正常终止进程来调试

Nginx手册阅读笔记 —— 性能调优

URL: [https://www.nginx.com/blog/tuning-nginx/]

Linux 配置

Backlog Queue

net.core.somaxconn
等待被Nginx接收的请求队列。 Nginx 接收请求的速度很快，所以这个值通常不需要很大，默认值也是比较低的。但如果你的网站负载很高，可以尝试增加这个队列。如果内核的错误日志显示这个值太小，可以提高这个值直到错误停止。注意： 如果你把这个值设置的大于512，调整Nginx listen指令的backlog参数来与这个值匹配。

net.core.netdev_max_backlog
在每个网络接口接收数据包的速率比内核处理这些包的速率快时，允许送到队列的数据包的最大数目。 如果你的带宽很高，建议调大这个值

File Descriptors

文件描述符是用来处理诸如连接或打开文件等事情的系统资源。Nginx最多为每个连接使用2个文件描述符。譬如做代理时，客户端请求占一个文件描述符，发起的后端请求会占一个。如果一个系统有大量的连接，下面的这些设置需要被调整：
sys.fs.file_max
系统全局的文件描述符限制

nofile
用户可使用的文件描述符限制，在 /etc/security/limits.conf文件中设置
理解它们的关系：

  • 所有进程打开的文件描述符数不能超过/proc/sys/fs/file-max
  • 单个进程打开的文件描述符数不能超过user limit中nofile的soft limit
  • nofile的soft limit不能超过其hard limit
  • nofile的hard limit不能超过/proc/sys/fs/nr_open

Ephemeral Ports

当使用Nginx作为代理时，每个向后端发起的请求都会使用一个临时的端口。
net.ipv4.ip_local_port_range
端口的取值范围。如果你的端口耗尽，可以修改这个范围。通常的值为 1024~65000

net.ipv4.tcp_fin_timeout
端口被另一个连接使用前的冻结时间。默认值为60秒，通常可以减少到30 到 15秒之间。

Nginx手册阅读笔记 —— 核心指令

URL: [http://nginx.org/en/docs/ngx_core_module.html]

accept_mutex on | off
如果开启，worker进程将轮流接收新请求。否则，有新请求到来时，所有的worker进程将集体被唤醒，而此时如果请求流量较低，一些worker进程只会浪费系统资源。 参考： [http://huoding.com/2013/08/24/281]

accept_mutex_delay time
启动accept_mutex后，每个worker进程重新接受新请求的最大时间间隔。默认值 500ms

debug_connection
对指定的客户端请求启动debug模式。其它请求使用error_log指令设置的日志级别。可以指定 IP/主机名/unix套接字。 这个指令的生效需要在编译时指定 —with-debug
